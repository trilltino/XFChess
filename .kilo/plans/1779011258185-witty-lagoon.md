# PGN Generation Implementation Plan

## Objective
Enable PGN (Portable Game Notation) generation for all chess games (P2P, Solana, Tournament) with logs available on the server and viewable in the admin panel.

## Current State
- No PGN generation exists anywhere in the codebase
- `moves` table has `move_san` column but it's never written (schema.rs:64)
- `MoveEvent` on-chain emits only UCI moves (moves_ix/record.rs:111-118)
- Backend stores `move_uci` only (repository.rs:289-298, add_move_simple)
- Admin panel has no PGN viewer

## Phase 1: Engine PGN Module (crates/nimzovich_engine)

### 1.1 Create src/pgn.rs
```rust
pub mod pgn;
```

Features needed:
- `square_to_algebraic(sq: i8) -> [char; 2]` — convert 0-63 to "e4" etc
- `piece_id_to_san_role(piece: i8) -> Option<&'static str>` — K/N/B/R/Q or None for pawn
- `is_en_passant(game: &Game, dst: i8) -> bool`
- `disambiguate(game: &Game, piece: i8, src: i8, dst: i8) -> Option<[char; 2]>`
- `move_to_san(game: &Game, src: i8, dst: i8, promo: i8) -> (String, Option<Suffix>)`
- `Suffix` enum (Check, Checkmate)
- `PgnDocument` struct with tags, moves Vec<String>, result String
- `PgnBuilder` that accumulates moves via `add_move(move_uci: &str)`
- `PgnBuilder::to_pgn_string() -> String`

### 1.2 Add to lib.rs
```rust
#[cfg(feature = "std")]
pub use pgn::{PgnBuilder, PgnDocument};
```

Gate behind `std` feature only — on-chain program stays unchanged.

### 1.3 Dependencies
- No new dependencies needed — use existing `constants.rs` piece IDs
- `heapless::String` for bounded allocations (or just `String` since std feature)

## Phase 2: Backend PGN Generation (backend/src)

### 2.1 New PGN generation module: backend/src/pgn.rs
```rust
pub struct PgnGenerator {
    engine: Game,  // nimzovich_engine::Game
}

impl PgnGenerator {
    pub fn new() -> Self;
    pub fn from_fen(fen: &str) -> Result<Self>;
    pub fn add_move_uci(&mut self, uci: &str) -> Result<San>;
    pub fn finalize(&self) -> PgnDocument;
}
```

### 2.2 Wire into record_move (backend/src/signing/routes/main.rs:264)
Currently: `repo.add_move_simple(...)` only stores UCI.

Add fire-and-forget task that also:
1. Loads game state from DB (or maintains in-memory)
2. Calls `engine.do_move()` for each move
3. Calls `move_to_san()` to generate SAN
4. Stores `move_san` in DB

But: storing SAN per move is light; full PGN generation at finalize is cleaner.

### 2.3 Wire into finalize_game (backend/src/signing/routes/main.rs:331)
After `repo.complete_game()` in the fire-and-forget:
1. Fetch all moves for the game: `repo.get_moves(game_id)`
2. Replay through `PgnGenerator`
3. Call `pgn.to_pgn_string()`
4. Store in games table: add `pgn_text TEXT` column

### 2.4 Database schema update (backend/src/db/schema.rs)
Add to games table:
```sql
pgn_text TEXT,
```

### 2.5 API endpoint: GET /games/{id}/pgn
New route in history.rs:
```rust
.route("/games/{game_id}/pgn", get(get_game_pgn))
```
Returns `{ "pgn": "..." }`

## Phase 4: P2P / Tournament Scenarios

### 4.1 P2P Games (direct connection)
- Same flow: moves recorded via /move/record → finalize → PGN generated
- No changes needed; PGN works the same

### 4.2 Solana On-chain Games
- Moves validated on-chain via `chess_logic_on_chain::validate_and_apply`
- Backend records moves via /move/record
- Same PGN generation at finalize

### 4.3 Tournament Games
- Tournament matches use same game flow
- Swiss tournament: each match finalized separately → PGN per match
- Single elimination: bracket matches → PGN per match
- Admin panel shows tournament bracket with PGN links

## Phase 5: PGN Logging / Server Visibility

### 5.1 Server-side PGN storage
- `games.pgn_text` column — persisted in SQLite
- Archived via existing archiver (tasks/archiver.rs)
- Optional: write PGN files to `/pgn/` directory for backup

### 5.2 Admin panel PGN access
- `GET /games/moves/{game_id}` — existing endpoint returns moves with SAN
- `GET /games/{game_id}/pgn` — new endpoint returns full PGN
- Admin panel renders PGN viewer

## Data Flow Summary

```
UCI move on-chain (MoveEvent)
  → /move/record POST (stores move_uci)
    → fire-and-forget: replay moves through nimzovich_engine
      → generate SAN per move
      → store move_san in moves table
  → /game/finalize POST
    → fire-and-forget: replay all moves → build PgnDocument
      → store pgn_text in games table
      → GET /games/{id}/pgn returns PGN
  → Admin panel calls GET /games/{id}/pgn
    → renders PGN viewer
```

## Phase 6: MagicBlock Move Log Links

### 6.1 Concept
Each move should have a link to the MagicBlock TEE endpoint where that move was processed/validated.

The MagicBlock ER (Execution Rollup) is at: `https://devnet-eu.magicblock.app/` (config.rs:70)
TEE endpoints: `/tee/pubkey`, `/tee/attestation`

### 6.2 Data Model
Add to `moves` table:
```sql
magicblock_tx_sig TEXT,    -- ER transaction signature for this move
magicblock_endpoint TEXT,  -- URL of the ER that processed this move
```

### 6.3 Implementation
In `record_move` (main.rs:204), after signing and submitting via ER:
1. Get the ER transaction signature (`sig`)
2. Store it with the move: `repo.add_move(..., magicblock_tx_sig: &sig, magicblock_endpoint: &state.config.er_rpc_url)`
3. This gives each move a verifiable link to the TEE that processed it

### 6.4 Admin Panel
- `GET /games/moves/{game_id}` returns moves with `magicblock_tx_sig` and `magicblock_endpoint`
- Admin can click the sig to view on Solana Explorer / MagicBlock
- PGN viewer shows move numbers with clickable links to their ER signatures

### 6.5 For Tournament Games
- Each match in a Swiss/elimination tournament is its own game
- Each move has its own ER signature
- Tournament admin panel shows bracket with PGN links, each move clickable

### 6.6 Verification Flow
1. Move recorded → signed via ER → signature stored
2. At any time, admin can: GET /games/{id}/moves → see all signatures
3. Each signature links to: `{er_rpc_url}/tx/{sig}` (MagicBlock explorer)
4. PGN includes move numbers; clicking move N opens ER tx for that move

## Implementation Order
1. Engine pgn.rs (lowest risk, pure Rust)
2. Backend pgn.rs + schema update
3. Wire into finalize_game
4. Add API endpoint
5. Admin panel PGN viewer
6. MagicBlock move log links (add columns, store sigs, expose in API)
